## 2. 평가지표

```
Score = 0.5 × (1 - NMAE) + 0.5 × FICR
```
- **NMAE**: `|예측 - 실제| / 설비용량`의 평균. 단, 실제 발전량이 설비용량의 10% 미만인 시간대는 평가에서 제외.
- **FICR**: 오차율 기준 계단식 단가 구조.
  - 오차율 ≤ 6% → 단가 4.0
  - 6% < 오차율 ≤ 8% → 단가 3.0
  - 오차율 > 8% → 단가 0.0
  - `FICR = sum(실제발전량 × 단가) / sum(실제발전량 × 4.0)` — 발전량이 큰 시간대일수록 그 시간의 오차가 점수에 더 크게 반영되는 가중 구조

### 10. 계절별 파워커브 - 풍속 구간 및 표본 수 기준 정밀 분석

풍속을 15구간으로 나누어 계절별 평균 발전량과 표본 수를 함께 확인했습니다.

- **중간 풍속(3\~10m/s대, 표본 수백\~수천 건)**: 세 그룹 모두 겨울이 봄가을·여름보다 뚜렷하게 높음 (예: 그룹1 4.5\~5.6m/s 구간에서 겨울 9,482 vs 여름 6,919). 공기밀도 가설과 일치하며, 표본이 충분하고 세 그룹에서 일관되게 나타나 신뢰할 만하다고 판단했습니다.
- **저풍속(~2m/s 이하)**: 봄가을이 오히려 높게 나오는 경우가 있으나, 발전량 자체가 작아 순위가 쉽게 뒤집히는 구간으로 판단했습니다.
- **고풍속(12m/s 이상)**: 겨울 표본이 한 자릿수(심지어 n=1)까지 떨어져, 겨울철 급락처럼 보이는 패턴이 실제 현상인지 표본 부족에 의한 노이즈인지 현재 데이터로는 판단할 수 없다고 결론을 내었습니다.
- 이 발견은 3번 섹션의 8%문턱 진단(오차가 중고출력 구간에 집중)과 겹치는 구간이라, 공기밀도를 feature로 명시적으로 추가할 근거가 됐습니다.

### 11. SCADA 이상치와 그룹 매핑 검증

info.xlsx로 추정한 매핑(VESTAS 1\~6호기=그룹1, 7\~12호기=그룹2)을 SCADA로 검증하려 했으나, 최초 상관계수가 그룹1 0.0013, 그룹2 -0.0001로 거의 무상관이었습니다. 그런데 같은 데이터의 시계열 그래프는 거의 겹칠 정도로 비슷해서, 이 모순의 원인을 추적했습니다.

이상치 제거 후 개별 터빈 상관계수를 재계산하니 wtg01\~06은 전부 그룹1과, wtg07~12는 전부 그룹2와 더 높은 상관관계(0.8\~0.93대)를 보여, 애초 매핑이 맞다는 게 확인했습니다. 최종 그룹 합계 상관계수는 그룹1은 0.9654, 그룹2는 0.9688입니다(그룹3은 애초부터 0.9489로 정상).

## 5. Feature Engineering

### Feature 목록

| feature | 근거 |
|---|---|
| `ldaps_ws_10m`, `gfs_ws_10m/80m/100m`, `ldaps_ws_50m_max_approx` | u,v 벡터를 풍속 크기로 변환. LDAPS 10m이 EDA에서 가장 강한 상관관계를 보여 우선 사용 |
| `*_std` (격자 표준편차, 특히 YBLWS, prmsl) | EDA에서 예상 밖으로 강한 신호(0.4~0.67)로 확인돼 처음부터 mean과 함께 집계 |
| `ws_diff_ldaps_gfs`, `_abs` | LDAPS-GFS 불일치 정도 (EDA에서 평균 2.6m/s 차이 확인) |
| `air_density` | 기압/기온 기반 이상기체 법칙 근사. 계절별 EDA에서 확인한 공기밀도 효과를 명시적 변수로 반영 |
| `powercurve_kpx_group_*` | train 라벨로 학습한 경험적 파워커브 (25구간), test엔 적용만 |
| `hour_sin/cos`, `month_sin/cos` | 계절성·시간대 패턴 반영 |
| `gfs_planetaryBoundaryLayer_0_VRATE_mean` | 유지 여부를 상관관계로 판단 후 포함 |

EDA에서 세제곱 변환이 파워커브의 포화구조로 인해 오히려 상관관계를 낮췄기 때문에, 세제곱(`wind_speed ** 3`) feature는 의도적으로 넣지 않았으며, 원본 풍속을 그대로 넘기고 트리 모델이 비선형을 학습하도록 했습니다.

최종 feature 개수: **150개**. `train_df`, `test_df`를 parquet으로 저장해 모델링 단계에서 재사용합니다.

### 공기밀도 계산 버그

공기밀도를 이상기체 법칙(`밀도 = 기압 / (R × 절대온도)`, R=287.05)으로 계산했는데, 최초 결과의 평균이 0.570kg/m³로 나왔습니다. 실제 공기밀도는 지표 부근에서 보통 1.0~1.3kg/m³ 범위라 정상 범위의 절반 수준이었습니다.

원인을 추적해보니 LDAPS 기온 변수가 이미 켈빈 단위로 제공되고 있었는데, 섭씨로 착각해서 코드에서 273.15를 한 번 더 더하고 있었습니다. 원본 기온값을 직접 확인하니 평균 281.17K(≈섭씨 8도)로, 태백 지역의 기후와 맞아떨어져 이미 켈빈이라는 것을 확인했습니다. `+ 273.15`를 제거하고 재계산하니 평균 1.125kg/m³로 정상 범위에 들어왔습니다.

### VRATE 판단

`gfs_planetaryBoundaryLayer_0_VRATE_mean` 변수는 대회 데이터 설명 문서에 "행성경계층 수직 속도"로 기재되어 있으나, NCEP GFS 공식 문서 기준으로 동일 변수명(VRATE)은 "환기율(Ventilation Rate, PBL 높이 × 풍속, 단위 m²/s)"로 정의됩니다. 
실제 관측값(0\~수만 범위)도 "수직 속도(m/s)"로 보기엔 물리적으로 불가능한 크기이며, PBL 높이와 풍속의 곱 구조와는 자연스럽게 일치해, 환기율 해석이 맞다고 판단했습니다.
행성경계층 환기율(VRATE) 변수는 분포가 치우쳐 있어 포함 여부를 판단해야 했다. 발전량과의 직접 상관관계는 0.376\~0.393(그룹별), 이미 만든 LDAPS 10m 풍속과의 중복도는 0.67이었습니다. 완전히 겹치는 수준은 아니라고 판단해(설명 안 되는 분산이 약 55% 남음) 그대로 유지했습니다.

### 데이터 누수 방지

- SCADA는 평가 기간 데이터 자체가 제공되지 않아 feature로 사용하지 않았습니다.
- 파워커브 feature는 train 라벨로만 학습하고, test에는 학습된 변환표를 적용했습니다.

## 6. Modeling

### 모델 비교 (default 파라미터)

FE로 만든 150개 feature를 그대로 사용해, 배깅 계열(RandomForest)과 부스팅 계열(XGBoost, LightGBM, CatBoost, HistGradientBoostingRegressor) 5개 모델을 각 라이브러리의 기본값으로 우선 비교했습니다. HistGradientBoosting은 sklearn 내장 모델로, 별도 설치 없이 LightGBM과 비슷한 히스토그램 기반 부스팅 성능을 낼 수 있어 비교군에 추가했습니다.

**로컬 검증(2024년 10~12월 hold-out) 결과:**

| 모델 | Score | 1-NMAE | FICR |
|---|---:|---:|---:|
| CatBoost | 0.61377 | 0.87300 | 0.35454 |
| XGBoost | 0.61218 | 0.86731 | 0.35705 |
| LightGBM | 0.61113 | 0.87357 | 0.34870 |
| HistGradientBoosting | 0.60919 | 0.87362 | 0.34475 |
| RandomForest | 0.59573 | 0.87201 | 0.31945 |

원본 베이스라인(FE 이전, 0.57759) 대비 전 모델이 개선되어, FE가 실제로 효과가 있었다는 것을 확인했습니다. 부스팅 계열이 배깅(RF)보다 뚜렷하게 나은 것도 확인했는데, 이는 부스팅의 순차적 오차 보정 방식이 파워커브의 비선형 포화 구조를 더 잘 잡아내기 때문으로 해석했습니다.

### 리더보드 제출 검증 — 로컬 검증의 한계 발견

5개 모델을 전부 제출해 실제 순위와 비교했다.

| 모델 | 로컬 Score | 리더보드 Score | 로컬 순위 | 리더보드 순위 |
|---|---:|---:|---:|---:|
| CatBoost | 0.61377 | 0.61090 | 1위 | **1위** |
| XGBoost | 0.61218 | 0.59997 | 2위 | **5위(최하위)** |
| LightGBM | 0.61113 | 0.60834 | 3위 | 2위 |
| HistGradientBoosting | 0.60919 | 0.60947 | 4위 | 3위 |
| RandomForest | 0.59573 | 0.60693 | 5위 | 4위 |

XGBoost가 로컬 2위에서 리더보드 최하위로, RandomForest가 로컬 최하위에서 리더보드 4위로 뒤바뀌었습니다. 기존 검증 기간(2024년 10~12월 3개월)이 계절이 한쪽으로 치우쳐 있어, 실제 test(2025년 전체, 사계절 포함)와 성격이 달랐던 것으로 판단했습니다.

### 검증 방식 재설계 — 계절 대표 fold

검증 기간을 "매년 1·4·7·10월(겨울·봄·여름·가을 대표)"로 재정의했다. lag feature(과거 시점 발전량에 의존하는 변수)를 사용하지 않아, 시간 순서를 무시하고 연도를 섞어 검증셋을 구성해도 데이터 누수가 발생하지 않는다는 것을 먼저 확인한 뒤 진행했습니다.

재검증 결과, 절대 점수는 전반적으로 낮아졌으나(계절 전환기 데이터가 예측 난이도가 높기 때문), **순위의 정확도는 개선됐습니다** — XGBoost가 정확히 최하위로, RandomForest가 정확히 중하위(4위)로 나타나 실제 리더보드 순위와 더 잘 맞아떨어졌습니다.

### 하이퍼파라미터 튜닝

Optuna로 계절 대표 3-fold 교차검증 기준 튜닝을 설계했습니다. 모델별 학습 시간 차이가 커서(RandomForest는 depth 제한이 없으면 1회 학습에 약 25분 소요) 아래와 같은 단계적 접근을 취했습니다.

**1단계 (5개 모델, 가벼운 탐색):**

| 모델 | 평균 Score (3그룹) |
|---|---:|
| HistGradientBoosting | 0.59138 |
| LightGBM | 0.58705 |
| XGBoost | 0.58572 |
| RandomForest | 0.58415 |
| CatBoost | 0.58266 |

상위 3개(HistGradientBoosting, LightGBM, XGBoost)를 2단계 심화 튜닝 대상으로 선정했습니다.

### CatBoost 결과 재검증

1단계 튜닝에서 CatBoost가 5개 모델 중 최하위로 나왔는데, default 파라미터로 실행했던 앞선 실험에서는 CatBoost가 오히려 1위였다는 게 마음에 걸렸습니다. 같은 모델이 이렇게까지 순위가 뒤집히는 게 이상하다고 판단해, default와 튜닝 결과를 동일한 검증 조건으로 다시 비교해보기로 했습니다.

재검증 과정에서, 튜닝된 파라미터를 다시 불러와 모델을 재구성할 때 일부 고정 설정(조기 종료 조건 등)이 누락되는 구조적 문제를 발견했습니다. 이를 수정해 다시 비교한 결과, CatBoost는 적은 시도 횟수(6회)로는 default 성능을 넘어서지 못한다는 것을 확인했습니다:

- default: 0.59970
- 튜닝 결과(수정 후 재현): 0.59255

CatBoost는 자체 기본값이 이미 여러 데이터셋에서 검증된 값으로 알려져 있어, 적은 시도 수로는 이를 능가하기 어려웠던 것으로 판단했습니다. 이후 비교에서는 CatBoost를 튜닝 대상에서 제외하고 default 파라미터 그대로를 후보로 유지하기로 했습니다.

### 튜닝 중단 및 방향 전환

2단계(상위 3개 모델 심화 튜닝) 진행 중, 모델별 default/1단계 성능이 이미 서로 근접해 있고(0.582~0.591 범위), Colab 무료 환경에서 CPU로 이 이상 탐색을 확장하는 것은 시간 대비 개선폭이 크지 않다고 판단해 이 시점에서 튜닝을 마무리하기로 했습니다. 트리 계열 모델은 여기까지의 결과(1단계 상위 3개 + CatBoost default)를 현재까지의 최선 후보로 유지하고, 남은 자원은 구조적으로 다른 접근(딥러닝 비교)에 투입하기로 했습니다.

트리 기반 부스팅(XGBoost, LightGBM, CatBoost 등)은 시계열 전용 기법이 아니라 범용 머신러닝 앙상블 기법이라는 점도 이 단계에서 다시 짚었습니다. 이 프로젝트에서 시계열 성격은 모델 구조가 아니라 캘린더 feature 설계와 계절 기준 검증 분리에 있으며, 시간 순서를 구조적으로 학습하는 딥러닝 계열(LSTM, GNN 등)과는 이 지점에서 근본적인 차이가 있습니다.
